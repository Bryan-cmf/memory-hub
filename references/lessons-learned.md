# Lessons Learned — 16 次實戰踩坑全集

> 來源：UltraClaw 記憶系統架構設計 v2.0 + 日常踩坑記錄

---

## 坑 #2 (2026-05-17)：三層防護同時失效

| 項目 | 內容 |
|------|------|
| **情況** | 下午連續 6 項任務後，Agent 忘記寫 daily log |
| **數據** | Layer 1（即時寫入）失效 → Layer 2（auto_sync）掃不到 → Layer 3（checkpoint）無法觸發 |
| **根因** | 任務密度太高，meta-task「寫記憶」被擠出注意力 |
| **教訓** | 致命鏈：L1 失效 → L2 跟著失效 → L3 無法定義觸發條件 |

## 坑 #25 (2026-05-18)：WhatsApp 跨 Session 記憶黑洞 🔴

| 項目 | 內容 |
|------|------|
| **時間** | 14:40-16:39 HKT |
| **Session** | 859ca859 (WhatsApp) |
| **數據** | 32 次 tool call、3 份 PDF（3.6MB）、**0 次 write** |
| **根因** | Agent 根本不知道要寫記憶——獨立 session，不繼承任何記憶規則 |
| **教訓** | 計數 ≠ 記錄。跨 session 規則不應依賴 Agent 繼承 |
| **解法** | JSONL 掃描作為主力層，系統級自動提取 |

## 坑 #5：config warning ≠ 實際狀態

| 項目 | 內容 |
|------|------|
| **情況** | 看到 config 警告就假設服務不可用 |
| **教訓** | 先查數據再下結論，config warning 可能只是格式問題 |

## 坑 #6：API 有限制時直接讀檔案系統

| 項目 | 內容 |
|------|------|
| **情況** | sessions_list API 有渠道限制，無法看到 WhatsApp session |
| **解法** | `find ~/.openclaw/logs -name "*.jsonl"` 直接讀檔案系統 |
| **教訓** | 跨渠道整合必須繞過 API 限制 |

---

## 坑 #8：跨時區日期匹配

| 項目 | 內容 |
|------|------|
| **情況** | UTC 日期和 HKT 日期不一致，造成 daily log 錯日 |
| **解法** | 三日期 OR 匹配：HKT + UTC + UTC-1 |
| **教訓** | 時區是分散式系統永恆的痛 |

---

## 坑 #10：跨渠道整合

| 項目 | 內容 |
|------|------|
| **情況** | sessions_list API 只能看到當前渠道的 session |
| **解法** | 直接讀取所有渠道的 JSONL 檔案系統目錄 |
| **教訓** | 不要依賴單一 API，檔案系統是終極真相 |

---

## 坑 #11：標記命名空間隔離

| 項目 | 內容 |
|------|------|
| **情況** | 多個 cron job 用同一個 marker，互相覆蓋 |
| **解法** | 每個 cron job 獨立命名空間的 marker |
| **教訓** | 共享狀態必須顯式隔離 |

---

## 坑 #12：Append 被 write tool 覆蓋

| 項目 | 內容 |
|------|------|
| **情況** | Agent 用 write tool 寫 daily log，覆蓋了之前的內容 |
| **解法** | 強制 `cat >>` append；auto_sync 不修改原文件 |
| **教訓** | Append 操作永遠用追加模式，禁止 write tool |

---

## 坑 #16：JSONL 格式雙版本兼容

| 項目 | 內容 |
|------|------|
| **情況** | OpenClaw 升級後 JSONL 格式變更，掃描器崩潰 |
| **解法** | try/except per line + v1/v2 adapter 模式 |
| **教訓** | 格式變更不可怕，可怕的是沒有 fallback |

## 坑 #17-#24：日常小坑精選

| # | 坑 | 教訓 |
|---|-----|------|
| 17 | 忘記在 Session 開始讀 MEMORY.md | 開局必讀記憶文件 |
| 18 | lessons 寫了但沒更新 index | 所有文件變更都要更新索引 |
| 19 | 項目記憶只寫了一次就不再更新 | 項目記憶需要持續維護 |
| 20 | daily log 格式不統一 | 嚴格按模板格式 |
| 21 | 忘記關聯相關的歷史記憶 | 發現關聯時主動記錄 See Also |
| 22 | 跨 session 的決策沒有記錄原因 | 記錄「為什麼」，不只是「做了什麼」 |
| 23 | 多次重複踩同一個坑 | 坑記錄必須被搜尋和回顧 |
| 24 | Agent 模型的記憶規則被新 prompt 覆蓋 | 持久規則放在 PERMANENT-RULES.md |

## 終極教訓

> **問題不在 Agent 記性不好。問題在記憶系統的核心輸入源是 Agent 手動操作。**
>
> **系統級掃描 > Agent 自覺。這是 MemoryHub 存在的根本理由。**
